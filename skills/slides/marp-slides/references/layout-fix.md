# layout-fix.md — 視覚検証ループ

Marp は Tailwind クラスの効き方や折り返し・オーバーフローをエディタ上のプレビューだけでは正確に判断できない。実際にレンダリングした PNG を目視して初めて崩れが見つかる。本ドキュメントは「PNG 出力 → 目視 → 修正 → 再出力」を崩れがゼロになるまで回す手順をまとめたもの。SKILL.md のワークフロー Step 6（視覚検証）から呼ばれるほか、既存デッキの特定スライドだけを整形したいとき（記事でいう `/layout-fix` 相当）にも単独で使える。

## 1. PNG 出力

```bash
marp deck.md --images png --html --allow-local-files
```

`--html --allow-local-files` は常に必須。付け忘れると `<script>`/`<style>` が無効化されたり、ローカル画像の読み込みがブロックされてスタイルが一切効かない（`setup.md` 参照）。

ブラウザが見つからずレンダリングに失敗する場合は、`CHROME_PATH` に環境にインストール済みの Chrome/Chromium バイナリのパスを指定する。

```bash
CHROME_PATH="/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" \
  marp deck.md --images png --html --allow-local-files
```

未設定でも動く環境もある。初回起動（コールドスタート）は 1〜4 分ほどかかることがあるので気長に待つ。

出力は `deck.001.png`, `deck.002.png` ... のようにスライドごとの連番ファイルになる。全スライド分を Read で開いて目視する。

## 2. 目視（Read）の検出基準

PNG を1枚ずつ開き、以下の観点で崩れがないか確認する。

| 検出基準 | 見るポイント |
| --- | --- |
| はみ出し（オーバーフロー） | 要素がスライド枠の外に出ていないか、下端で見切れていないか |
| 要素の重なり | テキストや画像・アイコンが別の要素に重なっていないか |
| テキスト切れ | 文言が途中で切れている、または `...` もなく省略されていないか |
| 余白バランス崩れ | 左右・上下の余白が極端に偏っていないか、パネル間の間隔が不揃いでないか |
| コントラスト不足 | 背景と文字色の明度差が足りず読みにくくないか（画像上のテキストは特に注意） |
| スタイル未反映 | em タイポや brand 色が全く効いておらず、素の Markdown 表示のままになっていないか |

最後の「スタイル未反映」が疑われる場合は、個別の崩れ修正ではなく後述の「5. フォールバック」に進む。

## 3. 修正パターン表

崩れの種類ごとに、まず以下の定番パターンを試す。独自の対処を考案する前に、必ずこの表から選ぶ。

| 症状 | 対処 |
| --- | --- |
| オーバーフロー | `gap-4`→`gap-2`、`p-12`→`p-8` など余白縮小、または本文を `text-em-base`→`text-em-sm` に段階ダウン |
| 縦5項目以上 | 2列グリッド（`grid-cols-2`）＋見出し行に `col-span-2` |
| パネル内テキストあふれ | 文言を短縮（要点だけ） |
| 画像上の文字が読めない | オーバーレイ不透明度を上げる（`bg-black/40`→`/55`） |

## 4. 反復

1つの修正を当てたら、その場で直感的に「直った」と判断せず、必ず再レンダリングして PNG を再度目視する。崩れがゼロになるまで「修正→再レンダリング→目視」を繰り返す。1回の修正で複数箇所が変わることもあるため、直したスライド以外に新たな崩れが出ていないかも合わせて確認する。

## 5. フォールバック

PNG で Tailwind クラスが一切効いていない（スタイル未反映）疑いがある場合、静止画のレンダリングだけでは原因切り分けが難しい。その場合は `marp -s .`（サーバモード）でデッキを起動し、ブラウザまたは Playwright でスクリーンショットを撮って確認する記事本来の方式に切り替える。DevTools のコンソールでスクリプトエラーが出ていないか、`--allow-local-files` の付け忘れがないかを合わせて確認する。

## 6. レンダリング時の注意点

- **`---` / `***` はスライド区切り**: Marpit では `---` や `***` を単独行に書くと新しいスライドの開始とみなされる。通常の Markdown の `<hr>`（水平線）としては使えないため、スライド内に区切り線を引きたい場合は `<hr>` タグか `border-t` 系のクラスを使う。
- **ヘッダー直後の `---` は空スライドを作る**: `templates/slide-header.md` の内容（frontmatter とスクリプト）をコピーした直後に区切りの `---` を置くと、内容のない空スライドが先頭に1枚できてしまう。最初の実スライドの内容はヘッダーと同じスライド内に続けて書くか、空白スライドができることを承知の上で意図的に区切る。
